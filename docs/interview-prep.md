# Job Radar 面试准备稿

## 1. 项目一句话

Job Radar 是一个面向 2027 届求职的个人招聘信息台。它把企业官网、国聘、央国企公告、高校就业网、牛客、实习僧、公众号和就业群里的招聘信息，整理成一个可筛选、可审核、可推送、可追踪投递状态的工作台。

它不是单纯爬虫项目，核心目标是把分散、低信噪比、容易错过的招聘情报，转成每天可以执行的求职动作。

## 2. 项目背景

### 解决的问题

27 届招聘信息分散在很多地方：企业官网、国聘、央国企公告、高校就业网、牛客讨论区、公众号、就业群、内推表格。人工每天刷会遇到几个问题：

- 信息源太多，容易漏掉关键批次，比如提前批、暑期实习、秋招、春招补录。
- 平台信息质量参差不齐，销售、生产制造、低相关岗位会污染视线。
- 公众号、群消息、牛客帖很有价值，但结构不统一，不能直接混进正式岗位库。
- 看到岗位之后没有后续管理，容易忘记截止时间、投递状态和下一步动作。
- 只看互联网大厂会导致样本偏差，非互联网、央国企、金融、快消、咨询、湖南/长沙本地机会容易被忽略。

### 目标用户

第一用户是我自己：2027 届、重点关注产品、策略产品、AI 产品、数据分析、数据科学、算法/机器学习、经营分析、决策支持等方向。

可扩展用户是同样处在秋招准备期的同学，尤其是需要同时关注互联网和非互联网机会的人。

### 用户场景

- 早上收到飞书新增机会摘要，先看今天有没有值得投的岗位。
- 打开信息台，按 27 届、产品/策略、算法/数据、非互联网、即将截止筛选。
- 对岗位标记状态：感兴趣、已投递、笔试、面试、Offer、忽略。
- 对群消息、公众号、牛客帖先进入审核池，核验公司、岗位、截止、官网链接后再进入主库。

## 3. 产品方案

### 核心链路

```text
信源配置
-> 抓取岗位
-> 字段标准化
-> 去重与增量合并
-> 规则打分和降噪
-> 信息台展示
-> 新增推送
-> 投递状态管理
```

### 信息台视角

信息台不是单纯列表，而是回答这些问题：

- 今天该看什么？
- 哪些是 27 届主线机会？
- 哪些岗位马上截止？
- 互联网和非互联网机会结构是否失衡？
- 哪些信源异常？
- 哪些线索还需要人工审核？

## 4. 技术选型

### 为什么没有用 Next.js

这个项目目前没有选择 Next.js，而是选择 Python + 静态 HTML + GitHub Actions。

原因是：

- 主要复杂度在数据采集、清洗、去重、调度和推送，不在复杂前端交互。
- 使用者主要是自己，第一版更需要稳定、低成本、免服务器，而不是搭一套完整 Web 应用。
- GitHub Pages 可以直接托管静态 HTML，配合 GitHub Actions 就能做到自动更新。
- 单文件 HTML 内嵌数据，打开快、部署简单、移动端也能看。
- 后续如果需要多人账号、服务端搜索、权限、在线编辑，再迁移到 Next.js 更合理。

如果面试官问“为什么不用 Next.js”，可以这样答：

> 我没有为了用框架而用框架。这个项目的一期核心问题是信息采集和决策流，不是复杂前端状态。静态 HTML 能用最低维护成本验证产品价值。等到出现多用户、权限、在线编辑、服务端搜索这些需求时，再引入 Next.js 会更自然。

### 为什么用 Python

- Python 标准库足够完成 CSV、JSON、HTML 导出、HTTP 请求、规则处理。
- 招聘信息处理更偏数据管道，Python 写规则、清洗、打分效率高。
- Playwright 可以补充处理牛客、实习僧、SPA 高校等浏览器渲染源。

### 为什么用 GitHub Actions

- 不需要本机开机。
- 不需要自己买服务器。
- 适合定时任务、轻量 ETL、静态站部署。
- 可以把数据文件 commit 回仓库，天然保留变更历史。

### 为什么用 GitHub Pages

- 零服务器部署。
- 适合静态信息台。
- 飞书推送里可以直接带在线链接。

### 为什么用 Feishu Webhook

- 推送能进入日常工作流，而不是要求用户每天主动打开页面。
- new-only 推送能避免刷屏。

## 5. 技术架构

### 目录职责

- `config/sources.csv`：信源配置，定义 source_id、adapter、endpoint、priority、fetch_method、status。
- `job_radar/adapters/`：各类网站抓取器。
- `job_radar/sync.py`：主同步逻辑，负责读取信源、抓取、标准化、去重、打分、增量合并。
- `scripts/sync_plan.py`：调度入口，区分 fast、slow、full、rescore。
- `scripts/export_html.py`：导出静态信息台。
- `scripts/notify_preview.py`：生成新增推送预览。
- `scripts/send_notify.py`：发送飞书/企微 webhook。
- `data/jobs.json`：当前岗位库。
- `data/jobs_archive.json`：下线/消失岗位归档。
- `data/notify_state.json`：已推送记录，避免重复推送。

### 定时任务

现在有三条 workflow：

- `job-radar-sync`：每天 07:00 跑 `fast`，抓稳定信源。
- `job-radar-slow`：每周三/周日 09:47 跑 `slow`，补牛客、实习僧、SPA 高校、飞书招聘等慢源。
- `job-radar-pages`：同步成功后部署 GitHub Pages。

### fast 和 slow 的区别

`fast` 跑稳定、轻量、高收益源：

- 国聘
- 国家平台
- 央企公告
- 大厂/外企官网
- 部分 HTML 高校源

`slow` 跑高成本源：

- 牛客
- 实习僧
- SPA 高校
- 飞书招聘
- 需要 Playwright 的源

这样做的原因是：每天全量抓取太慢，也更容易触发限流；但只跑 fast 又可能漏掉慢源高价值信息。所以拆成主链路和慢源补扫。

## 6. 数据处理逻辑

### 抓取

每个信源在 `config/sources.csv` 里配置 adapter。同步时按 source_id 选择本次要跑的源，跳过 blocked/deprecated 信源。

adapter 输出统一结构的 RawJob：

- title
- company_name
- location
- publish_time
- deadline
- official_url
- jd_text
- raw

### 标准化

RawJob 会被转换成 Job，补齐：

- source_id
- org_type
- industry
- salary
- dedup_key
- match_score
- tags
- risk_flags

### 去重

用 dedup_key 做主去重，不同来源的相似岗位会尽量合并，避免同一个岗位刷屏。

### 增量合并

- 新 dedup_key：新增岗位，写入 first_seen。
- 已有 dedup_key：刷新字段，保留 first_seen。
- 本次成功刷新源里消失的岗位：认为下线，移入 jobs_archive.json。
- 本次没跑的源：旧数据保留，不误删。

### 推送去重

`notify_state.json` 记录已经推送过的岗位。只有未推过、且满足筛选规则的新增岗位才会进入飞书推送。

## 7. 排序与筛选策略

项目不是简单按发布时间排序，而是结合用户画像做打分：

- 27 届优先。
- 产品、AI 产品、策略产品、数据科学、数据挖掘、算法/ML、决策支持优先。
- 非互联网、央国企、金融、快消、电商、咨询等方向单独看。
- 即将截止岗位提高优先级。
- 生产制造、泛销售、代招、劳务派遣、低相关管培等降权或标风险。

这样解决的是“岗位很多但不知道先看哪个”的问题。

## 8. 已达到的结果

可以这样讲，不要夸大：

- 搭建了一个完整的招聘信息台 1.0，从抓取、清洗、去重、打分、展示到推送形成闭环。
- 支持稳定信源每日自动快扫，慢源每周补扫。
- 支持 Feishu/企微 new-only 推送，避免重复刷屏。
- 支持 GitHub Pages 在线访问，不需要服务器。
- 支持下线岗位归档，当前岗位库和历史消失岗位分离。
- 支持半结构化信息审核，避免牛客帖、公众号、群消息直接污染主库。
- 支持投递状态看板，把信息发现转化成求职动作。

## 9. 遇到的困难

### 1. 信源质量和格式差异大

不同平台有 API、HTML、SPA、公告、讨论帖等不同形态。解决方式是 adapter registry：每种类型写一个 adapter，统一输出 RawJob。

### 2. 慢源和稳定性冲突

牛客、实习僧、SPA 高校需要浏览器渲染，运行慢且可能被限流。解决方式是拆 fast 和 slow workflow，主链路不被慢源拖垮。

### 3. 推送不能刷屏

如果每次运行都发全量，会造成信息疲劳。解决方式是 new-only 推送，并用 notify_state.json 记录已推送岗位。

### 4. 下线岗位如何处理

如果直接保留会污染当前信息台；如果直接删除又失去复盘价值。解决方式是当前库只保留在架岗位，下线岗位进入 jobs_archive.json。

### 5. 信息台容易偏向互联网大厂

早期抓到的互联网岗位多，导致首页被腾讯、字节等刷屏。解决方式是增加非互联网 tab、信源打散、行业雷达、央国企/金融/快消/咨询等单独视角。

## 10. 关键取舍

### 不做全自动导入公众号/群消息

原因是半结构化信息容易误识别，且来源可信度不一。选择先进入审核池，人工确认后再入主库。

### 不做每天全量抓取

原因是慢、容易限流、稳定性差。选择每天 fast，定期 slow。

### 不做复杂后端

原因是 1.0 目标是个人使用和验证价值，静态站已经够用。

## 11. 反思与下一步

### 当前不足

- 信源覆盖仍不均衡，金融、快消、电商、咨询、湖南/长沙本地机会还需要继续补。
- 部分慢源抓取稳定性仍受外部网站影响。
- 去重还可以更精细，比如同公司同岗位多城市、多来源重复。
- 推送内容还可以更个性化，比如按“今天必须投、值得看、只做情报”分层。
- 目前没有多用户、权限和在线编辑能力。

### 下一步

- 增加更多高价值非互联网信源。
- 做 RSS/公众号/就业群的半自动导入增强。
- 给信息台增加“今日必投”更明确的行动分层。
- 增加周报：本周新增、本周截止、本周投递进度。
- 如果多人使用，再考虑迁移到 Next.js + 数据库。

## 12. 面试官可能关心的问题

### Q1：这个项目和普通爬虫有什么区别？

普通爬虫只解决“抓到信息”。这个项目解决的是“从信息到行动”：抓取后还要去重、打分、降噪、分层展示、推送、投递跟踪和下线归档。

### Q2：为什么不用 Next.js？

因为 1.0 的核心复杂度在数据管道，不在前端框架。静态 HTML + GitHub Pages 已经能满足个人信息台的需求，维护成本更低。如果后续做多人账号、在线编辑、权限和服务端搜索，再引入 Next.js。

### Q3：为什么不用数据库？

当前数据规模和使用场景下，JSON 文件足够，而且 Git 仓库天然保存历史版本。数据库会增加部署和维护成本。等到多人协作、复杂查询、并发编辑出现，再迁移数据库。

### Q4：怎么避免重复推送？

发送成功后把岗位 key 写入 `data/notify_state.json`。下次生成推送时会过滤已推送岗位，所以同一岗位不会重复进入飞书。

### Q5：怎么判断新增？

每条岗位有 dedup_key。同步时和旧的 jobs.json 对比：旧库没有的新 key 就是新增；已有 key 就刷新；本次成功刷新源里消失的 key 进入下线归档。

### Q6：如果外部网站结构变了怎么办？

每个信源是独立 adapter，单源失败不会阻断整体同步。健康报告会记录失败次数、错误信息和抓取数量异常，便于定位是哪一个源坏了。

### Q7：为什么拆 fast 和 slow？

因为慢源通常需要 Playwright，运行慢且容易受限流影响。主链路每天稳定快扫核心源，慢源每周补扫或手动触发，兼顾覆盖率和稳定性。

### Q8：怎么保证信息质量？

通过多层机制：岗位方向打分、低质标签、风险标记、半结构化信息审核、下线归档、非目标岗降权。

### Q9：如果你要把它产品化，会怎么做？

我会优先做三件事：

1. 多用户画像：不同用户关注不同城市、行业和岗位方向。
2. 在线审核台：公众号、群消息、牛客帖可以在网页上确认后入库。
3. 周报/行动提醒：从信息发现进一步走到投递管理和复盘。

### Q10：这个项目体现了什么能力？

- 能把模糊需求拆成可运行系统。
- 能做数据链路、自动化调度和推送闭环。
- 能根据用户目标设计排序、筛选和降噪策略。
- 能在技术复杂度和产品价值之间做取舍。
- 能处理真实世界里不稳定、脏数据、多来源、重复信息的问题。
